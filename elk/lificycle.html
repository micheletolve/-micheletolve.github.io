Lificycle indici

Lifecycle indici

E' possibile gestire in modo automatico il ciclo di vita degli indici in modo da ottimizzare l'uso delle risorse (dico, memoria).

Tramite l'interfaccia grafica di kibana sotto la voce management>Index Lifecycle Policies abbimao definito tre policies, una per ogni tipologia di indice. Le policies definite non agiranno sugli indici fino, sarà necessario dopo la definizione applicarle agli indici già creati e a quelli futuri.

Ogni policies prevede diverse fasi nelle quali, con cadenze temporali o legate alla grandezza degli indici, è possibile passare da una fase all'altra nel seguente ordine: hot, warm, cold (con la possibilità di freezare l'indice) e delete.

La polcie "log" non prevede rollover (attività che spezza l'indice se troppo vecchio o troppo grande) poichè ogni giorno viene creato un nuovo indice per i log, mentre per altri tipi di indici il rollover viene gestito tramite l'uso degli alias. In questo modo chi dovrà mandare i dati lo farà puntando allo stesso nome che farà riferimento ad indici diversi.

Dopo aver definito le policies è possibile applicarle tramite il dev tools presente in kibana, ecco la lista di comandi necessari.

Restituisce la lista delle polocies definite

GET /_ilm/policy

Applica la policie "Log" a tutti gli indici che inziano con filebeat-*

PUT filebeat-*/_settings
{
    "lifecycle.name": "Log"
}

Per applicare la policie ai nuovi indici che verranno creati nel tempo è necessario definire un template tramite il seguente comando

PUT _template/filebeat_template
{
  "index_patterns": ["filebeat-*"],
  "settings": {
    "index.lifecycle.name": "Log"
  }
}

Se si vilesse modificare il lifecycle (modificando il numero di giorni o altre impostazione), tramite questo comando viene eliminata l'associazione con la vecchia definizione. Quindi dopo la modifica è possibile ripetere i comandi che abbiamo visto per applicare le policies agli indici.

POST filebeat-*/_ilm/remove

Come detto per i log raccolti da filebeat ogni giorno viene creato un indice quindi non ci sarà bisogno di tecniche di rollover. La stessa cosa non vale per "remote" o "transactionlog". Di seguito la lista di comandi che servono a definire un alias per questi indici e associare politiche di rollover tramitre le lifecycle policies

poichè si tratta di un indice già creato senza alias lo metto temporaneamente in sola scrittura

PUT /remote/_settings
{
  "settings": {
    "index.blocks.write": true
  }
}

sposto tutti i dati presenti in un indice temporaneo

POST _reindex
{
  "source": {
    "index": "remote"
  },
  "dest": {
    "index": "temp-remote"
  }
}

elemino l'indice "remote" perchè altrimenti avrei elastic si rifiuterà di chiamare un alias con lo stesso nome di un indice già presente

DELETE /remote

applico al pattern la lifecycle definita indicando anche il nome dell'alias

PUT _template/remote_template
{
  "index_patterns": ["remote-*"],
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 1,
    "index.lifecycle.name": "Remote",
    "index.lifecycle.rollover_alias": "remote"
  }
}

creo l'indice con l'alias associato

PUT remote-000001
{
  "aliases": {
    "remote": {
      "is_write_index": true
    }
  }
}

riempio l'indice con i dati presenti nell'indice temporaneo

POST _reindex
{
  "source": {
    "index": "temp-remote"
  },
  "dest": {
    "index": "remote-000001"
  }
}

ultimo step, elimino l'indice temporaneo

DELETE /temp-remote

Stesso procedimento per il transactionlog.
